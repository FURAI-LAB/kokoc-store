## 1. Валидация количества в корзине (критично)

**Было:** `POST /api/cart/items` брал `qty` из тела запроса и клал в БД без проверок,
хотя `PATCH` того же ресурса проверял. Проверено эксплойтом:

| Запрос | Старое поведение |
|---|---|
| `qty: 999999` | 201 OK, корзина на 4 989 995 010 ₽ при остатке 5 |
| `qty: 2.7` | 201 OK, дробное количество в БД |
| `qty: -5` | необработанный 500 (CHECK-констрейнт) |

**Стало:** общий валидатор `parseQty()` для обоих хендлеров.
`Number()` + `Number.isInteger` вместо `parseInt` (тот принимал `"12abc"` как 12).
Склад проверяется по **итоговому** количеству строки, а не по дельте — иначе
«добавить 2» трижды обходило остаток 5.

Файлы: `src/routes/api/cart.js`

## 2. Stored XSS через загрузку изображений (критично)

**Было:** `file.type` (полностью под контролем загружающего) писался в R2 как
`contentType`, а прокси `/r2/` отдавал его обратно. Файл с `type: "text/html"`
выполнялся как HTML на вашем домене. Расширение бралось из `file.name` — тоже без проверок.

**Стало:** `lib/uploads.js` — расширение **и** content-type берутся из фиксированного
whitelist, `file.type` только сверяется на противоречие. SVG запрещён (может нести `<script>`).
Применено в обоих местах загрузки (товары + коллабы).

Дополнительно закрыто в прокси:
- `sanitizeR2Key()` — блокирует traversal и мусорные ключи;
- принудительный `nosniff` (ветка возвращала голый `Response` мимо `withSecurityHeaders`);
- **уже загруженные файлы** с опасным content-type отдаются как `application/octet-stream`.

Файлы: `src/lib/uploads.js`, `src/routes/admin/products.js`, `src/routes/admin/index.js`, `src/server.js`

## 3. CSRF только на корзине

**Было:** origin-проверка жила внутри `cart.js` и защищала только `/api/cart/*`.
`/api/orders`, `/api/minigame/*`, `/api/subscribe`, отзывы — были доступны cross-origin.

**Стало:** `lib/csrf.js`, гард поднят в `routes/api/index.js` — новые роуты защищены
по умолчанию, а не по факту вспоминания.

Заодно закрыта дыра, найденная при переносе: старое `referer.startsWith(origin)`
пропускало `https://kokoc.store.evil.com`.

Файлы: `src/lib/csrf.js`, `src/routes/api/index.js`, `src/routes/api/cart.js`

## 6. Объединение /crocs и /adidas

**Было:** два файла по ~1250 строк, ~84% идентичных; блок разметки — побайтово одинаковый.

**Стало:** `pages/brand-catalog.js` (одна реализация) + `config/brand-pages.js` (данные).
`crocs.js` → 18 строк, `adidas.js` → 17 строк. Сигнатуры не изменились.

**Важная находка по ходу:** сначала я собрал общий модуль из `crocs.js` — и это
молча выбросило размерные таблицы одежды (`SIZE_GUIDE_APPAREL`, пол × верх/низ),
которые были только у Adidas. Тесты этого не поймали, поймало сравнение
отрендеренного HTML. Модуль пересобран из Adidas как надмножества.

**Побочно исправлен старый баг:** клиентский код в `crocs.js` рисовал 4 заголовка
(`US/UK/EU/CM`) для взрослых строк, где всего 3 значения — в таблице размеров
в quick-view была лишняя пустая колонка.

Проверка эквивалентности: заголовки, meta description, canonical, H1, чипсы
фильтров, заголовки и строки таблиц размеров, FAQ, количество JSON-LD блоков —
**побайтово совпадают** со старой версией для обоих брендов и обеих локалей.

`catalog.js` намеренно не тронут: у него своя логика `brand`-параметра в URL
и пагинации, сливать его тем же движением рискованно.
